# Claude Code 多 Agent 编排 · 设计原理(参考资料)

> **这份文档讲的是 Claude Code(Anthropic 官方源码)里多 Agent 编排到底怎么设计、怎么运转。**
> 它是**参考资料**,不是本项目(agent-orchestra)的设计说明——本项目正是照着这套原理来复刻。
>
> 重点是**讲清原理和架构思路**,代码只在必要处点一下(注明它在源码哪个文件),不堆大段看不懂的源码。
> 想看精确到行号的源码深拆 → 见 Claude Code 仓库 `docs/agent-orchestration.md`。

---

## 0. 先记住这一句

> **Claude Code 没有"多 Agent 调度器"。它只有一个会「自己调用自己」的主循环。**
> 派一个子 Agent,本质就是"带着一份干净的上下文,再跑一遍这个主循环"。

整篇文档都是在解释这句话。理解了它,就理解了 Claude Code 的多 Agent 编排。

---

## 1. 它是 "Agent" 不是 "Workflow"(最根本的设计选择)

业界有个经典二分:

- **Workflow(工作流)**:流程是**人写死**的——"先做 A,再做 B,然后 C"。像流水线。
- **Agent(自主体)**:流程是**模型每一轮临场决定**的——它自己看情况决定下一步。像一个会思考的人。

**Claude Code 是后者。** 这是它一切设计的根。

为什么重要:写代码、查 bug、做调研这类任务是**开放的**,你不可能为每种情况预先画好流程图。唯一可行的办法是——**让模型自己规划**:要不要读文件、读哪个、要不要派助手去查、够了没。

> 落到源码:这个"自主"就体现在主循环 `src/query.ts` 的 `while(true)` 里——每一轮都重新问模型"下一步干什么",没有任何写死的流程在指挥它。

---

## 2. 主循环:整个系统的心脏

Claude Code 的核心是一个循环(源码 `src/query.ts` 的 `queryLoop`),用大白话描述就是:

```
循环:
  ① 想:把当前对话发给模型,问"下一步做什么?"
  ② 模型如果只是回话、没要求用工具 → 任务结束,输出答案
  ③ 做:执行模型要求的工具(读文件、搜索、派 Agent……)
  ④ 看:把工具结果塞回对话里
  ⑤ 回到 ①(模型基于新结果,再决定下一步)
```

**三个要记住的原理:**

- **自主性就在第 ① 步**:每一轮都是模型自己决定干什么,不是程序安排的。
- **"结束"是模型自己说了算的**(第 ②步):模型不再要求用工具 = 它觉得做完了。**没人规定"走到第几步停"**——这正是 Agent 和 Workflow 最直观的差别。
- **这天然就是"链式推理"**:结果塞回去→再想→再做,一环扣一环。所谓 "Prompt Chaining(串行链)"模式,在这里是循环的**自然形态**,不是单独写的功能。

---

## 3. 工具编排:为什么能"又快又不出乱子"

模型一轮里常常要求执行**好几个**工具。怎么并发才安全?

Claude Code 的设计(源码 `src/services/tools/toolOrchestration.ts`)极简而聪明:

> **让每个工具自己声明"我能不能和别人同时跑",而不是让调度器去猜。**

```
每个工具自带一个标签:
   只读类(搜索、读文件)→ 可并发
   写入类(改文件)      → 不可并发

调度规则:
   连续的只读工具 → 合成一批,同时跑(有上限,默认 10 个)
   遇到写入工具   → 它自己单独一批,排队一个个跑
```

打个比方:**读资料像在图书馆,十个人可以同时查;写文件像在一块白板上写字,一次只能一个人,否则会写花。**

**为什么这么设计:**
- **同时拿到"快"和"安全"**:读的事并行(快),写的事排队(不互相覆盖)。
- **调度器不用懂每个工具**:它只看一个"能不能并发"的标签,把"懂不懂自己"的责任交给工具本身。
- **边跑边出结果**:不是等一批全跑完才返回,而是谁先好谁先返回——所以你能看到多个任务的进度同时滚动。

> 记住这条规则,下一节会看到它带来一个"白捡的好处"。

---

## 4. 子 Agent:多 Agent 的真相 = 递归

**这是整篇最重要的一节。**

当任务太大,模型会派一个"子 Agent"去办一块。但子 Agent **不是另写的一套系统**。看源码会发现一个关键事实:

> 派子 Agent 的代码(`src/tools/AgentTool/runAgent.ts`),最后调用的是 `query()`——**和主对话用的是同一个主循环函数。**

也就是说:

```
主 Agent 在主循环里
  └─ 模型决定:"用'派 Agent'这个工具"
       └─ 系统做三件事:
            ① 给子 Agent 一份干净的上下文(只带它要办的子任务)
            ② 再跑一遍主循环 ← 就是第 2 节那个循环,递归!
            ③ 子 Agent 办完,只把【最终结论】交回给父级
```

**三个要记住的原理:**

- **没有"子 Agent 框架"**:它就是主循环的递归调用。懂了主循环 = 懂了多 Agent。
- **"干净上下文"是为了保护父级的脑子**:子 Agent 自己烧预算去翻 50 个文件,回来只交一句结论。父级的对话不会被一堆中间过程刷屏——**这就是为什么和 Claude 聊一整天,它也不会越聊越糊涂**。脏活都被子 Agent 在外面消化了。
- **并行是"白捡"的**:"派 Agent"这个工具,本身被标记成"可并发"(第 3 节的规则)。所以模型一次派出多个子 Agent 时,它们**自动落入"只读并发批",一起跑**——Claude Code **没有为"多 Agent 并行"单独写过任何代码**,它是第 3 节那条规则的副产品。

> 这就是第 3 节说的"白捡的好处":一条"读并发"的规则,顺手就实现了"多 Agent 并行"。**这是整个架构最优雅的地方。**

## 5. 两种协作方式:项目经理制 vs 同事群

子 Agent 是"父派子"。当多个 Agent 要**协作**时,Claude Code 有两套并列的方式(按任务选用,不是先后顺序)。

### 5.1 项目经理制(协调器 / 星型)

源码:`src/coordinator/coordinatorMode.ts`

```
        👔 leader(项目经理:只派活 + 综合,自己不下场)
       ╱      │      ╲
      🛠      🛠      🛠   ← worker(各自是一个子 Agent,并行干活)
       ╲      │      ╱
        结果汇报给 leader,leader 综合后回复用户
```

三个设计要点:
- **角色分离**:leader 只拆任务、派活、综合、跟用户沟通,**不自己写代码**。
- **工人权力被收窄**:worker 拿不到"再派 Agent / 组队"的能力——**编排权牢牢留在 leader 手里**,工人只管干活。这样星型结构才稳。
- **最妙的一招——结果"伪装成用户消息"回灌**:worker 是异步的,办完后,它的结果被包装成一条"看起来像用户又说了句话"的消息塞回给 leader(源码里叫 `<task-notification>`)。
  **为什么这么做**:这样 leader 的主循环**完全不用为"等后台任务"写特殊逻辑**——它只是"又收到一条消息",照常进入下一轮。异步协作就这样无缝融进了同一个主循环。

> 这就是教科书里的 **Orchestrator-Workers(编排-执行)模式**。

### 5.2 同事群(邮箱 / 网状)

源码:`src/utils/mailbox.ts`(整个文件才 70 多行)

```
   🧑‍💻 ◄──── send ────► 🧑‍💻
   teammate 之间直接发消息,谁需要谁喊一声
```

两个设计要点:
- **通信靠一个极简邮箱**:就三个动作——`send`(发),`receive`(收,没消息就挂起等),`poll`(瞅一眼)。**通信机制本身极简,复杂度在"谁能给谁发"的上层。**
- **必须显式发消息**:一个 Agent 的普通输出,别的 Agent **看不见**;要沟通**必须**调"发消息"这个动作。这条强约束让**每次协作都有记录、可追踪**。

### 5.3 怎么选

| 用项目经理制(星型) | 用同事群(网状) |
|---|---|
| 任务有明确主理人,要集中控制和综合 | Agent 之间要对等协作、互相校验 |
| 控制力强、好追溯 | 灵活、自组织 |

> 一句话:**星型 = 项目经理 + 工人;网状 = 几个同事拉个群。**

## 6. 安全带:自主 ≠ 失控

能自己决定下一步的 Agent,也可能陷入死循环、无限派 Agent、停不下来还在烧钱。所以 Claude Code 把"刹车"当成内置设计:

| 刹车 | 防什么 | 怎么做 |
|---|---|---|
| **轮次上限(maxTurns)** | 主循环 `while(true)` 转太多圈 | 超过上限就强制结束(源码 `query.ts`) |
| **递归深度守卫** | Agent 派 Agent 无限套娃 | 嵌套太深就拒绝再派 |
| **级联取消** | 用户喊停后,后台 Agent 还在空跑 | 取消信号传给所有衍生的 Agent,一起停 |

> 这三道刹车不是补丁,而是"敢让它自主"的前提。

## 8. 设计精华一页纸

| 设计 | Claude Code 怎么做 | 一句话为什么 |
|---|---|---|
| 范式 | 自主 Agent,不是写死的 Workflow | 开放任务没法预先编排,只能让它自己规划 |
| 调度 | 没有中央调度器 | 协作从简单规则里"长出来",不靠聪明的调度 |
| 主循环 | 一个 `while(true)`,每轮问模型 | 自主性的载体;结束由模型自决 |
| 并发 | 工具自报"能不能并发" | 又快又安全;还顺手实现了多 Agent 并行 |
| 多 Agent | 递归调用同一个主循环 | 不造新框架,懂主循环就懂多 Agent |
| 上下文 | 子 Agent 给干净上下文、只回传结论 | 保护父级的脑子,长跑不变笨 |
| 协作 | 星型(经理制)+ 网状(同事群)两套 | 集中控制 vs 对等协作,按需选 |
| 通信 | 必须显式发消息,普通输出互不可见 | 协作可追踪,不靠"自言自语" |
| 安全 | 轮次上限 + 递归守卫 + 级联取消 | 自主的前提,不是事后补丁 |

## 9. 这套设计里,经典的"五大编排模式"在哪

很多人学过《Building Effective Agents》里的五大模式。它们在 Claude Code 里**不是五个独立功能,而是这套自主架构里自然出现的位置**:

| 五大模式 | 在 Claude Code 里它"长"在哪 |
|---|---|
| Prompt Chaining(串行链) | 主循环:结果回灌成下一轮输入(第 2 节) |
| Routing(路由) | 模型选派哪个子 Agent / 哪个工具(第 4、5 节) |
| Parallelization(并行) | 工具"读并发"分批(第 3 节) |
| Orchestrator-Workers(编排-执行) | 协调器:leader 派 worker(第 5.1 节) |
| Evaluator-Optimizer(评估-优化) | 用现有机制组合:一个 Agent 产出 → 另一个评判 → 再改 |

> 结论:Claude Code **没有刻意去"实现五大模式"**,但因为自主骨架搭得对,五大模式自然都在里面了。这就是"好的原语 > 聪明的调度"。
