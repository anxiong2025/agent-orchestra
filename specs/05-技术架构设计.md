## 1. 一句话架构

> **一个「可递归、可隔离、可流式」的主循环,加一套「读并发 / 写独占」的分批规则。多 Agent 协作不是另造的系统,而是「递归地再跑这个循环」+「两套通信拓扑」。**

这是 Claude Code 的核心设计哲学(其 `docs` §6),也是本项目的脊柱。**不存在一个叫 `Orchestrator` 的中央调度器。**

## 2. 分层总览

```
┌─────────────────────────────────────────────────────────────┐
│  业务层(皮肤,可替换)                                        │
│  M11 供应商挖掘+画像   ·   未来:竞品分析 / 市场调研 ...        │
├─────────────────────────────────────────────────────────────┤
│  协作层(多 Agent)                                            │
│  M5 子Agent(递归)  M6 协调器(星型)  M7 mailbox(网状)      │
├─────────────────────────────────────────────────────────────┤
│  内核层(单 Agent 的心脏)                                     │
│  M4 主循环  ──驱动──  M3 工具编排(读并发/写独占·流式)        │
├─────────────────────────────────────────────────────────────┤
│  原语层(地基,人人依赖)                                      │
│  M1 message · model(抽象+Mock)   M2 tool · context           │
├─────────────────────────────────────────────────────────────┤
│  适配层(provider,可灵活切换)                                │
│  M9 Bedrock/Claude · OpenAI/GPT · 国内模型 —— 都实现同一 Model │
└─────────────────────────────────────────────────────────────┘
```

**分层的铁律:依赖只能向下,不能向上。**
- 内核(主循环/编排)**只认原语层的抽象**(Message / Tool / Model),不知道用的是哪家模型、是不是子 Agent。
- 协作层是"在内核之上递归调用 + 加通信",**不修改内核**。
- 业务层是最薄的一层"皮肤",换场景只动这一层。

> 这层纪律是本项目能"学得清"的根本:**任何一层都能单独读懂,不必先理解上面或下面。**

## 3. 模块依赖图

```
                       cli.py(进度看板 / 未来的运行入口)
                          │
        ┌─────────────────┼───────────────────┐
        ▼                 ▼                   ▼
   coordinator.py     subagent.py         mailbox.py        ← 协作层
        │                 │                   │
        └────────┬────────┘                   │
                 ▼                             │
              loop.py ──────────────────────── │            ← 内核:主循环
                 │                             │
                 ▼                             │
          orchestration.py                     │            ← 内核:工具编排
                 │                             │
        ┌────────┼────────┐                    │
        ▼        ▼        ▼                    ▼
     tool.py  context.py  ───────────► message.py           ← 原语层
        │                                  ▲
        ▼                                  │
     model.py(Model 抽象)─────────────────┘
        │
        ▼
   providers/*(M9:Bedrock / OpenAI / 国内,均实现 Model)   ← 适配层
```

读法:**箭头 = "依赖/调用"**。注意 `loop.py` 既被 `subagent.py` 调(子 Agent 递归),又调 `orchestration.py`(执行工具)——这条"自己被自己的衍生物调用"的回路,就是递归编排的本质。

## 4. 三条关键数据流

### 4.1 单 Agent:ReAct 循环(Reason → Act → Observe,M4 主循环)

```
messages ──► model.complete() ──► 助手回应(可能含工具调用)
   ▲                                      │
   │                                      ▼
   │                            有工具调用?──否──► 结束,产出最终答案
   │                                      │是
   │                                      ▼
   └──── 工具结果回灌 ◄──── orchestration.run_tools()(M3)
                                  │
                          partition:读并发 / 写独占
```

> **Prompt Chaining 就在这里涌现**:工具结果回灌成下一轮输入,"链"是循环的副产品,不是单独的功能。

### 4.2 工具编排:一轮里的并发决策(M3,⭐ 最核心)

```
模型一轮吐出: [Read, Read, Write, Read]
                  │
        partition_tool_calls()  (按 is_concurrency_safe 切连续批)
                  │
     ┌────────────┼────────────┐
     ▼            ▼            ▼
 [Read,Read]    [Write]      [Read]
  并发 gather    独占 await    并发
 (Semaphore≤10)              边完成边 yield
```

> **多 Agent 能并行,是这条规则的副产品**:子 Agent 的 `AgentTool` 标记为 `is_concurrency_safe=True`,于是它落入"只读并发批",自动并行——没有为"多 Agent 并发"写过任何专门调度。

### 4.3 多 Agent:递归 + 两套拓扑(M5/M6/M7)

```
              主 Agent (loop.py)
                   │ 调用 AgentTool(它是个工具)
                   ▼
        ┌──── 子 Agent = 再调一次 loop.py ────┐   ← M5 递归 + 上下文隔离
        │   (干净上下文,depth+1,只回传结论)  │
        └──────────────────────────────────────┘

   两种协作拓扑:
   ┌─ 星型 (M6 协调器) ─┐      ┌─ 网状 (M7 mailbox) ─┐
   │   leader            │      │   A ◄──send──► B    │
   │  ╱  │  ╲            │      │   │  ╲   ╱  │       │
   │ W   W   W           │      │   C ◄──────► D      │
   │ 只对 leader 汇报     │      │  teammate 直接通信   │
   │ (task-notification) │      │  (SendMessage 工具)  │
   └─────────────────────┘      └─────────────────────┘
```

## 5. 五个支撑全局的设计决策(及理由)

| 决策 | 怎么做 | 为什么(对标 Claude Code) |
|---|---|---|
| **模型可注入** | `Model` 抽象 + `MockModel` | 编排无需 API key 即可跑通/测试;换 provider 只加类。对标 `query/deps.ts` 依赖注入 |
| **provider 无关接口** | `Model.complete` 只用内部 Message/ToolCall | 多模型(Claude/GPT/国内)灵活切换的根;格式归一化关在 provider 层 |
| **声明式并发** | 工具自报 `is_concurrency_safe` | "并发"由数据声明,不靠调度器猜;多 Agent 并行自然涌现。对标 `Tool.ts` |
| **流式生成器** | 全链路 `async generator` + `yield` | 边跑边出结果,多 Agent 进度可同时观察。对标全篇 `async function*` |
| **自主必配安全带** | `max_turns` + 递归 `depth` 守卫 | 给了多步自由,就必须给走不出去时的强制刹车。对标 `query.ts` maxTurns / `isInForkChild` |

## 6. 边界:复刻什么,不复刻什么

**复刻(编排的灵魂)**:主循环、读并发/写独占、子 Agent 递归与隔离、协调器、mailbox 通信、熔断与递归守卫。
做完这些 = 一个"五脏俱全"的 Claude Code 编排内核。

**刻意不复刻(生产规模的工程外壳)**:6 种任务后端、远程 bridge、Perfetto 追踪、实验开关、5 层上下文压缩、prompt cache 字节级优化、swarm 的布局/重连/权限同步。
理由:这些是**工程复杂度**,不是**编排原理**;全复刻会淹没核心,违背"最透明、原理最清"的项目护城河。

> 一句话:**复刻 Claude Code 多 Agent 编排的「机制与哲学」,而非「全部工程代码」。** 对"学懂原理 + 做 MVP",前者才是要的。

## 7. 演进方向(架构如何随迭代生长)

```
现在  ── 原语 + 内核(M1–M4):一个能 ReAct(推理→行动→观察)、会并发、能熔断的单 Agent
  │
  ├─► 协作(M5–M8):递归出子 Agent → 协调器 → 通信 → 评估优化循环
  │
  ├─► 真实化(M9–M10):多 provider 模型 + 调研工具(搜索/抓取)+ 溯源防幻觉
  │
  └─► 业务(M11):供应商挖掘+画像 —— 第一张"皮肤"
        未来换皮肤(竞品/市场调研)只动业务层,内核不变
```

**架构的成功标准**:加业务场景时只动最上层;换模型时只动最下层;**中间的编排内核一行不改**。如果某次改动逼你动了内核去迁就业务或某家模型,说明抽象漏了——那一刻就是回头修边界的信号。
