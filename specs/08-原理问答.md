## 总纲:一句话讲清整个项目

> "我用 Python 从零复刻了 Claude Code 的多 Agent 编排核心。**最大的收获是一个反直觉的事实:多 Agent 不是一个新框架,而是单 Agent 主循环的三种用法——递归调用自己(子 Agent)、星型派活(协调器)、网状通信(mailbox)。** 整个系统没有'中央调度器',协作行为是从'一个可递归的循环 + 一套工具分批规则'里**涌现**出来的。"

记住这张映射表(常被问到"五大 Agent 模式你怎么实现的"):

| 模式 | 在本项目哪里 | 是"写出来的"还是"涌现的" |
|---|---|---|
| Prompt Chaining(串行链) | M3 主循环 | 涌现:工具结果回灌成下一轮输入 |
| Routing(路由) | M5 子 Agent | 涌现:模型自决派哪个/几个子 Agent |
| Parallelization(并行) | M4 读并发 | 涌现:工具标 safe → 落入只读并发批 |
| Orchestrator-Workers | M6 协调器 | 字面实现:leader 派活 + 综合 |
| Evaluator-Optimizer | M8 | 组合现有零件,非新机制 |

## M1–M2 · 模型抽象 + 记忆

### Q · 你说"模型是无状态的",那多轮对话的"记忆"是怎么来的?

**机理:** 模型每次 `complete(messages)` 调用是**独立**的,它不会在两次调用之间"记住"任何东西。所谓"记忆",纯粹是**客户端这边把历史攒起来、每轮一起发过去**的结果。本项目里:

- 上下文 = 一个只增的 `list[Message]`(`message.py` 的 `Message`)。
- `loop.py` 每轮做三件事:append 用户输入 → 把**整个** messages 发给模型 → append 模型回复。
- 第二轮之所以"记得"第一轮,是因为第二轮把第一轮也一起发出去了。

**速记版:** "模型无状态,记忆是客户端维护的——上下文就是一个只增的 messages 列表,每轮把完整历史一起发给模型,它才'看得到'上文。记忆的本质就是'把历史重新发一遍'。"

### Q · 为什么把"调模型"抽象成一个接口(依赖注入)?

**机理:** `Model` 是抽象基类,只有一个 `complete(messages, tools)` 方法。真实跑用 `BedrockClaudeModel`,测试用 `MockModel`(按脚本返回,不烧 API)。编排层(loop/orchestration/subagent)只依赖 `Model` 这个抽象,**完全不知道**背后是 Bedrock 还是 Mock。换模型 = 换一个 provider 类,编排代码零改动。对标 Claude Code 的 `deps.ts`。

**速记版:** "把'调模型'做成可注入的抽象接口,真实跑注入 Bedrock,测试注入 MockModel——同一套编排代码两边通吃,CI 不烧 API。多 provider(M9)就是再加一个实现类的事。"

## M3 · ReAct 主循环

### Q · 什么是 ReAct?你的主循环为什么能"自己决定何时停"?

**机理:** ReAct = Reason(想)→ Act(做)→ Observe(看)的循环。`loop.py` 的 `run_agent_turn`:

```
while True:
    reply = await model.complete(messages)   # 想
    if not reply.tool_calls: return           # 看:模型不要工具了 = 它觉得做完了 → 停
    跑工具 → 结果 append 回 messages           # 做 + 把观察回灌
    # 回到顶:模型基于新结果再决策
```

**"何时停"由模型自决**:它这一轮只说话、不调工具,就等于宣告"做完了"。**没有任何代码规定"第几步停"**——这正是 Agent(自主)和死板 Workflow(写死流程)的本质区别。

**速记版:** "ReAct 是个 while 循环:调模型→若它要工具就执行并把结果回灌→再调模型。结束由模型自决(它不再要工具=做完了),没人写死步数。这就是 Agent 和 Workflow 的区别。"

### Q · 既然模型自决,怎么防止它陷入死循环烧钱?

**机理:** 配 `max_turns` 熔断(默认 10)。每轮递归前检查轮数,超限就注入一条"已达最大轮数"消息并强制结束。**能自主决策的东西,必须配安全带**——这是 M3 的核心教训,后面 M5 的 `max_depth` 是同一类思路。

**速记版:** "自主必须配熔断。max_turns 限制单个 Agent 转几圈,超了强制停。能自己决策就可能停不下来,安全带是必须的。"

## M4 · 读并发 / 写独占(⭐ 最该吃透)

### Q · 模型一轮要调多个工具,你怎么决定哪些并发、哪些串行?

**机理:** **不让调度器去猜,而是让每个工具自己声明 `is_concurrency_safe`**(只读=True,写=False)。`orchestration.py` 的 `partition_tool_calls` 按这个标签把一轮的工具调用切成**连续批次**:

- 连续的只读工具 → 合成一批,`asyncio.gather` 并发跑(`Semaphore(10)` 限流)。
- 写工具 → 自己独占一批,串行 await。

**注意是"连续"切分,不是"把所有读归一批"**:`[读,读,写,读]` → `[[读,读],[写],[读]]`。写工具会**打断**只读批,保住"读→写→读"的先后语义(否则把后面的读提到写前面就乱了)。

**速记版:** "工具自报 is_concurrency_safe,调度器只按这个标签切连续批:连续只读合一批 gather 并发(信号量限 10),写工具独占一批串行。声明式并发——调度不猜,工具自己说。"

### Q · 并发跑的工具结果,顺序乱了会不会出问题?

**机理:** 不会。多个工具结果可以按**任意顺序**回传给模型——provider 用 `tool_call_id` 配对工具和结果,**不靠位置**。所以"谁先跑完先 yield"(完成顺序 ≠ 原始顺序)对模型完全无害。这也是为什么 `_run_read_batch` 能用 `as_completed` 边完成边 yield。

**速记版:** "无所谓。工具结果靠 tool_call_id 配对,不靠顺序,所以并发批里谁先好谁先回都行。"

## M5 · 子 Agent:递归 + 上下文隔离

### Q1 ⭐ 上下文隔离是怎么做到的?(最高频)

**先破除误解:隔离没有任何"复杂机制",它是一个朴素事实的推论。**

**机理三步:**

**① "上下文"在本系统里 = 一个 `messages` 列表。** 这是 M2 就定下的抽象:对话历史是只增的 `list[Message]`。"模型记得什么"100% 取决于本次 `complete(messages)` 喂进去的那个列表——**模型本身无状态**,全靠你把历史一起发过去。

**② 隔离 = 给子 Agent【新建一个空列表】,而不是把父级的列表传给它。** `subagent.py`:

```python
sub_messages: list[Message] = [        # ← 全新的、独立的 list 对象
    Message.system(SUBAGENT_SYSTEM),   #   只装子 Agent 的工作守则
    Message.user(task),                #   + 这一个子任务
]
result = await run_agent_turn(sub_messages, model, sub_registry, child_ctx, ...)
```

父级的 `messages`(用户聊过啥、之前的工具结果与回复)**一条都没抄进来**。模型只看到这两条,**物理上无法感知**父级聊了什么。隔离来自**"不继承"**(这里"没做什么"),而不是某个"隔离器"。

**③ 这是"值隔离",不是"引用共享"。** `sub_messages` 是独立 list 对象,子 Agent 往里 append 的所有东西都进它自己这个列表,父级的 `messages` 一个字节都不变。两个列表在内存里是两个对象——这就是隔离的物理保证。

**出口也要干净——只回传结论。** 入口干净还不够:子 Agent 跑完,它的列表里攒了一大堆(系统提示+子任务+好几轮工具记录)。若整串塞回父级,父级照样被污染。所以 `run()` 结尾只取最后一条有正文的 assistant 消息:

```python
conclusion = _final_conclusion(result_messages)  # 只取最终结论
return conclusion                                 # 中间过程随局部变量被回收
```

父级的 `messages` 只多了一条"工具结果 = 那句结论",不是几十轮记录。

**隔离换来的价值(要点出价值):**
- **保护父级上下文窗口**:子 Agent 烧自己的几千 token 探索,父级脑子里只多一句结论,不会因子任务过程提前撑爆 context。
- **保护父级注意力**:无关中间步骤不进父级,综合时不被噪音干扰。

> **速记版:** "上下文在本系统里就是一个 messages 列表,模型无状态、靠把历史一起发才有记忆。隔离 = 给子 Agent 新建一个只含'系统提示+子任务'的独立列表,不继承父级历史(入口干净);跑完只 return 最后一条结论、丢弃中间过程(出口干净)。两个 list 是独立对象,父子互不污染。好处是保护父级的上下文窗口和预算。"
>
> 有测试钉死:`test_subagent_context_is_isolated` 在父级藏一句"机密",断言子 Agent 看到的 messages 里完全没有它。

### Q2 ⭐ "多 Agent"你是不是写了个新框架?子 Agent 怎么实现的?

**机理:没有新框架。子 Agent = 拿一份隔离上下文,再调一次主循环 `run_agent_turn`。** `AgentTool` 本身是个普通工具(和 read_file 平级),它的 `run()` 结尾就是 `await run_agent_turn(sub_messages, ...)`——和主 Agent 跑的是**完全相同的代码**,只是换了个干净的 messages 和 depth+1 的上下文。**递归复用主循环,就是多 Agent 的全部秘密。** 对标 Claude Code 的 `runAgent.ts` 结尾再调 `query()`。

> **速记版:** "没写新框架。AgentTool 是个普通工具,它的 run() 结尾就是递归调主循环 run_agent_turn——子 Agent 和主 Agent 跑同一段代码,只是上下文隔离、深度+1。多 Agent 就是主循环的递归复用。"

### Q3 ⭐ 模型怎么决定派几个子 Agent?能 2、3、4 吗?

**机理:派几个不是代码写死的,是模型自决的——Routing 模式涌现。** 分两个维度:

- **宽度(一次派几个)**:`spawn_agent` 是普通工具,模型一轮回复能吐多个工具调用,所以能一次派 2、3、4…个。**派几个 = 模型判断"任务能拆成几块独立子任务"**。"对比 Python/Rust"→ 判断 2 块 → 派 2 个。代码不规定数量。唯一硬约束是 M4 的 `Semaphore(10)`:派再多也最多 10 个同时跑,其余排队(全会跑完)。
- **深度(套几层)**:子 Agent 能否再派孙 Agent。这才是 `max_depth` 管的,且默认发生不了(见 Q4)。

> **速记版:** "派几个是模型自决的(Routing 涌现),取决于任务能拆成几块独立子任务,2/3/4 都行。代码不写死数量;只有 M4 信号量限制'同时在跑'最多 10 个。嵌套层数才由 max_depth 管。"

### Q4 ⭐ 怎么防止过度拆分 / 引导"何时该派、何时别派"?(实战踩坑)

**背景:** 第一版只问"对比 Python 和 Rust",模型却炸出 11 个 Agent(主1+子2+孙8),还反复调 write_file 死循环、单轮 60 秒。**根因不在代码逻辑,在'给模型的规则没拦它'。**

**机理:靠"代码约束 + prompt 纪律"两手,核心是把'派分身'这把锤子从子 Agent 手里拿掉(治本)。** 吸取 Claude Code 源码的**三道防线**:

1. **工具白名单(治本)**——`AgentTool(allow_nesting=False)` 默认从子 Agent 工具集里**剔除 `spawn_agent`**。**没有锤子就不会找钉子**,从源头杜绝套娃,根本不依赖深度计数。对标 `ALL_AGENT_DISALLOWED_TOOLS`。
2. **Prompt 纪律**——子 Agent 系统提示明写"自己直接完成,别再派"。对标 `Do NOT spawn sub-agents; execute directly`。
3. **"何时不该派"指导**——`spawn_agent` 描述写"读文件/查时间这类小事自己用工具更快"。对标 `When NOT to use the Agent tool`。

`max_depth` 退居"兜底护栏":仅当显式 `allow_nesting=True` 时才靠 depth 拦无限递归。

**配套修死循环:按任务收窄工具(`subagent_tools` 白名单)。** 子 Agent 被配了用不上的 write_file,纯总结任务却反复写文件,单轮飙到 60 秒。修法:研究类任务只给只读工具(`read_file`/`now`),不给 write_file。对标 `ASYNC_AGENT_ALLOWED_TOOLS`。再加"结论 300 字以内"治啰嗦(对标 `under 500 words`)。

> **速记版:** "防过度拆分主要不靠深度计数,而靠最干脆的一招——默认从子 Agent 工具集里剔除 spawn_agent,它根本没有派分身的能力。再加两道 prompt 纪律:系统提示让它'自己干完别再派',工具描述告诉主 Agent'简单的事别派'。另外按任务收窄子 Agent 的工具(纯研究任务不给写工具),治了'反复写文件死循环'那个 bug。"

### Q5 ⭐ 子 Agent 多个调用为什么能自动并行?你写并发代码了吗?

**机理:没写。并行是"工具声明 + M4 分批规则"的副产品。** `AgentTool.is_concurrency_safe = True` → 多个 `spawn_agent` 调用落入 M4 的"只读并发批" → `asyncio.gather` 自动并发(`Semaphore(10)` 限流)。**多 Agent 并行不是单独写的,是 M4 分批规则的副产品。**

> **速记版:** "一行没写。AgentTool 标了 concurrency_safe=True,多个子 Agent 调用就落进 M4 的只读并发批被 gather 并发跑——并行是分批规则的副产品,不是新机制。"

### Q6 · 你踩坑后总结的方法论是什么?(软实力题)

**机理:** 两条:
1. **多 Agent 可靠性 = 一半代码约束(工具白名单/深度守卫) + 一半 prompt 纪律(系统提示/工具描述)。** "模型派几个、调哪些工具、说多少话",调的都是工具集 + 提示词,不是编排逻辑。
2. **遇到"慢/怪"先加 per-turn 计时日志定位,别猜。** 当时一加日志,"反复写文件、单轮 60 秒"立刻现形(`scripts/diag_m5_timing.py`)。

> **速记版:** "多 Agent 的坑大多不在逻辑,在'给模型的规则/工具/提示'。可靠性一半靠代码约束、一半靠 prompt 纪律。另外调试黄金法则:遇到慢或怪,先加 per-turn 日志定位,别靠猜。"

## 附:可能被追问的"边界题"

| 追问 | 一句话答 |
|---|---|
| 子 Agent 失败了怎么办? | 当前 `run()` 返回结论字符串,失败信息也作为结论回灌;级联取消靠 `RunContext.abort`(M5 已埋,D2/D3 深做)。 |
| 父级 abort 了,子 Agent 会停吗? | 会。`child_ctx` 通过 `ctx.child()` 派生,**共享同一个 abort 信号**;`run_tools` 每批前检查 `abort.is_set()`,被叫停就不再开新批。 |
| 深度守卫和 max_turns 区别? | max_turns 限"单个 Agent 转几圈"(纵向时间),max_depth 限"Agent 套几层"(纵向递归)。两个不同维度的安全带。 |
| 为什么用工厂 `build_registry` 而不是直接传 registry? | 每个子 Agent 要拿**独立一份**工具实例(互不干扰),也方便按任务收窄(M6 给 worker 收权限)。 |
