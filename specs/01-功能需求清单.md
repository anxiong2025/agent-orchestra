# 功能需求清单(FRD)· 多 Agent 编排引擎

> **这份文档的双重身份:**
> 1. **产品需求**:列出本项目要实现的全部功能。
> 2. **复刻验收清单**:每条功能 = Claude Code 一个真实编排机制。**全部 ✅ = 几乎完全复刻 Claude Code 多 Agent 编排核心(含进阶)。**
>
> 覆盖尺度:**核心机制 + 流式 / Fork / 压缩等进阶**(生产级外壳如远程桥接/追踪/实验开关不在范围,见末尾"范围边界")。
> 每条都标注:对标的真实源码 `文件:行号`(可回 `claude-code-main` 核对)+ 验收标准。
> 实现顺序见 `07-迭代开发计划.md`;这份只管"要哪些功能、做了没"。

---

## 如何使用这份清单

- `[ ]` 未做 / `[x]` 已做。每条 = 一个独立可验收的功能。
- **P0** = 核心机制(不做就不算复刻);**P1** = 进阶机制(做了才"几乎完全复刻")。
- 每条的"✅ 验收"是判断"做对没"的硬标准。

## A · 基础原语(地基)

### [ ] FR-A1 · 消息模型 `P0`
- **要实现**:对话的数据结构;助手消息能带工具调用;工具结果能作为消息回灌。
- **对标**:`types/message.ts`
- **✅ 验收**:能构造 user/assistant/system 消息,助手消息可携带多个工具调用 + id。

### [ ] FR-A2 · 模型抽象 + 可注入 `P0`
- **要实现**:`Model` 抽象接口 `complete(messages)`;`MockModel`(脚本化,无需 API)。
- **对标**:`query/deps.ts`(依赖注入)
- **✅ 验收**:编排逻辑能在 MockModel 下完整跑通,零 API key。

### [ ] FR-A3 · 多 Provider 适配(可灵活切换) `P0`
- **要实现**:Claude(AWS Bedrock,开发期先做)/ GPT / 国内模型,均实现同一 `Model` 接口;`make_model(provider,...)` 工厂;各 provider 内部做 **消息/工具调用格式归一化**。
- **对标**:`query/deps.ts` 的可替换设计思想
- **✅ 验收**:改一个配置项即切换模型,编排代码零改动;Bedrock Claude 真实调通。

### [ ] FR-A4 · 工具抽象 + 并发安全自声明 `P0`
- **要实现**:`Tool` 接口(name/description/run);**关键字段 `is_concurrency_safe`**(只读 True / 写 False);工具注册表。
- **对标**:`Tool.ts`(`isConcurrencySafe`)
- **✅ 验收**:工具能注册、按名取用;并发安全性由工具自报。

### [ ] FR-A5 · 运行上下文 + 递归深度 `P0`
- **要实现**:`RunContext` 贯穿一次运行,含 `abort` 信号 + `depth`(递归层数);可派生子上下文(depth+1)。
- **对标**:`ToolUseContext` + `queryTracking.depth`(query.ts:347)
- **✅ 验收**:子上下文 depth 正确递增;abort 可沿上下文传播。

## B · 单 Agent 核心循环

### [ ] FR-B1 · 主循环(ReAct:推理→行动→观察) `P0`
- **要实现**:`while True` 异步生成器:调模型 → 无工具调用则结束 → 执行工具 → 结果回灌 → 下一轮。
- **对标**:`query.ts` `queryLoop`(行 241、307)
- **✅ 验收**:MockModel 脚本化"调工具→看结果→回答"能完整跑完;结束由"模型不再调工具"触发。
- **附带涌现**:Prompt Chaining(串行链)。

### [ ] FR-B2 · 工具分批:读并发 / 写独占 `P0` ⭐
- **要实现**:`partition_tool_calls` 按 `is_concurrency_safe` 切连续批;只读批并发(`Semaphore`,上限默认 10)、写批串行;边完成边 yield;contextModifier 在批次边界统一合并。
- **对标**:`toolOrchestration.ts`(行 91 分批、行 10 上限、行 42-62 合并)
- **✅ 验收**:`[Read,Read,Write,Read]` 分成 `[读读][写][读]`;两个 Read 实测同时开始(时间戳验证),Write 独占。
- **附带涌现**:Parallelization(并行)。

### [ ] FR-B3 · maxTurns 熔断 `P0`(安全带)
- **要实现**:每轮递归前检查轮数,超限注入提示并强制结束。
- **对标**:`query.ts:1705`
- **✅ 验收**:让 MockModel 永远调工具,能被 maxTurns 刹停。

## C · 多 Agent 协作核心

### [ ] FR-C1 · 子 Agent = 递归 + 上下文隔离 `P0` ⭐⭐
- **要实现**:`AgentTool`(本身是工具,`is_concurrency_safe=True`);run 内:造干净上下文(depth+1)→ **递归调用主循环** → 只回传最终结论。
- **对标**:`runAgent.ts`(行 248;行 748 递归调 query)
- **✅ 验收**:主 Agent 派子 Agent,子 Agent 独立跑完只回传结论;**派两个子 Agent 自动并行**(因落入只读并发批)。

### [ ] FR-C2 · 递归深度守卫 `P0`(安全带)
- **要实现**:depth 超上限拒绝再派子 Agent。
- **对标**:`forkSubagent.ts:78`(isInForkChild 思路)
- **✅ 验收**:递归超限时拒绝派生,不会无限套娃。

### [ ] FR-C3 · 子 Agent 上下文裁剪 + 权限收紧 `P0`
- **要实现**:子 Agent 不继承父级全部历史(只带子任务);异步子 Agent 自动避免权限提示。
- **对标**:`runAgent.ts:404-410, 440-451`
- **✅ 验收**:子 Agent 上下文显著小于父级;只回传结论不污染父级。

### [ ] FR-C4 · 协调器模式(星型 / Orchestrator-Workers) `P0` ⭐⭐
- **要实现**:开关开启后主 Agent 变 leader(只派活+综合);worker 工具白名单收窄(去掉"再派 Agent/组队");worker 结果包成**伪用户消息**回灌 leader。
- **对标**:`coordinatorMode.ts`(行 36 开关、行 29 白名单、行 144 task-notification)
- **✅ 验收**:leader 派 2 worker 并行 → 收 task-notification → 综合;worker 调编排工具被拒。
- **附带涌现**:Orchestrator-Workers + Routing。

### [ ] FR-C5 · 结果回灌(伪用户消息) `P0`
- **要实现**:异步 worker 结果包装成"看起来像用户消息"塞回主循环,无需为"等待异步"写特殊逻辑。
- **对标**:`coordinatorMode.ts:144-160`(`<task-notification>` 格式)
- **✅ 验收**:leader 主循环把 worker 结果当作"新到的一条消息"照常处理。

### [ ] FR-C6 · Agent 间通信:mailbox(网状) `P0` ⭐
- **要实现**:mailbox 队列 `send/receive/poll`;`SendMessage` 工具(Agent 间唯一通信方式,普通输出互不可见)。
- **对标**:`mailbox.ts`(行 33/54/48)+ `SendMessageTool/`
- **✅ 验收**:A `send` → B `receive` 挂起被唤醒;跑通"A 让 B 查、B 回传"。

## D · 进阶机制(做了才"几乎完全复刻")

### [ ] FR-D1 · 流式边收边执行 `P1` ⭐
- **要实现**:模型还在流式输出时,工具调用一冒出来就开始执行(不等整轮吐完)。
- **对标**:`StreamingToolExecutor.ts:40`;主循环 query.ts:658-863
- **✅ 验收**:能观察到"模型尚未输出完,首个工具已在跑"。
- **难点提示**:状态机 queued→executing→completed→yielded;结果仍按收到顺序 emit。

### [ ] FR-D2 · 兄弟级联取消 `P1`
- **要实现**:`siblingAbortController`(主 abort 的子节点);一个并行工具出错 → 立刻杀同批兄弟,但**不**向上中止主查询。
- **对标**:`StreamingToolExecutor.ts:48-61`
- **✅ 验收**:并行批里一个工具抛错,其余兄弟立即被取消;主循环不结束。

### [ ] FR-D3 · 三层 abort 级联 `P1`
- **要实现**:根 abort → 子 Agent(同步共享/异步独立)→ 兄弟工具,三层单向传播。
- **对标**:根/`runAgent.ts:706`/`StreamingToolExecutor.ts:59`
- **✅ 验收**:取消根 → 所有衍生 Agent 停;同步子 Agent 随父停、异步子 Agent 独立。

### [ ] FR-D4 · Fork 子 Agent(继承上下文 + 缓存友好) `P1` ⭐
- **要实现**:省略 subagent_type 触发;继承父级**完整**上下文 + 系统提示;占位符工具结果使前缀字节一致(prompt cache 复用);递归守卫防 fork 套 fork。
- **对标**:`forkSubagent.ts`(行 58 字节复用、行 107 构造、行 78 守卫)
- **✅ 验收**:多个 fork 子 Agent 前缀一致(可命中缓存);fork 内不能再 fork。

### [ ] FR-D5 · 上下文压缩(多层) `P1`
- **要实现**:至少做 micro(改写工具结果,缓存友好)+ auto(超阈值整段摘要);压缩发生在"调模型之前"。
- **对标**:`query.ts:413(micro)/454(auto)`;`services/compact/`
- **✅ 验收**:长对话超阈值时自动摘要,继续跑不崩;压缩点在调模型前。
- **可选加层**:snip / contextCollapse / reactive(应急)。

### [ ] FR-D6 · 工具结果预算裁剪 `P1`
- **要实现**:聚合工具结果过大时按预算裁剪。
- **对标**:`query.ts:379-388`
- **✅ 验收**:超大工具结果被截断到预算内,不撑爆上下文。

### [ ] FR-D7 · 评估-优化循环(Evaluator-Optimizer) `P1`
- **要实现**:用现有零件组合:产出→批判打分→不达标回灌重做→收敛/到上限。
- **对标**:组合机制(非单一源文件)
- **✅ 验收**:能看到"产出→批判→改进"收敛过程;有上限防死循环。
- **附带涌现**:五大模式最后一个。

## E · 任务后端 + 真实化 + 业务

### [ ] FR-E1 · 异步任务后端抽象 `P1`

## 范围边界(明确不复刻)

以下是 Claude Code 的**生产级工程外壳,非编排机制**,本清单**不含**(全做会淹没核心、几万行量级、与"学习/讲清原理/MVP"目标冲突):

- 远程桥接(跨机器 bridge)、远程 SSH worker 后端
- 进程内队友(InProcessTeammate)等多种任务后端
- Perfetto 链路追踪、GrowthBook 实验开关
- 提示缓存的字节级深度优化(Fork 我们只做"原理级"复刻)
- 跨会话 uds/bridge 寻址、团队布局/重连/权限同步

> 一句话:**复刻的是「机制与原理」,不是「全部生产代码」。这 24 条做完,讲得清原理、能演示、能落地 MVP——这就是"几乎完全复刻"的准确含义。**
