# Agent Orchestra · 迭代任务清单

> 一个**学习导向**的多 Agent 编排引擎。严格对标 **Claude Code 的真实编排设计**,用 Python 重写。
> 目标不是"又一个框架",而是 **每个迭代独立可跑、可读,讲清一个编排原理**。
>
> 用法:从 M1 往下,一个迭代一个 commit。每个迭代有【目标】【对标】【要写】【做完的标准】【学到什么】。
> 代码你来写;卡住了就拿着对应的"对标"去 Claude Code 仓库 `docs/agent-orchestration.md` 找那一节。

---

## 设计总原则(贯穿始终,来自 Claude Code §6)

> **不造中央调度器。编排 = 一个「可递归、可隔离、可流式」的循环 + 一套分批规则,让协作从规则中"长出来"。**

所以下面不是"堆功能",而是 **同一个核心循环,逐步长出能力**。先把"一个 Agent 怎么转"做对(M1–M4),多 Agent 不过是"递归地再跑这个循环"(M5+)。

### 和《Building Effective Agents》五大模式的关系

五大模式**不是**五个要单独造的功能,而是骨架搭对后**自然涌现**的位置:

| 模式 | 涌现在哪个迭代 |
|---|---|
| Prompt Chaining(串行链) | M4 主循环(结果回灌即链) |
| Routing(路由) | M5/M6(选派哪个子 Agent / 工具) |
| Parallelization(并行) | M3(读并发分批) |
| Orchestrator-Workers(编排-执行) | M6(协调器) |
| Evaluator-Optimizer(评估-优化) | M8(用现有零件组合,非新机制) |

### ✅ M0 · 项目骨架(已完成)
- uv 包 `agent-orchestra` 已建,Python 3.14。
- 你的起点:`src/orchestra/`。

---

### ⬜ M1 · 消息 + 模型抽象(让"对话"和"调模型"成型)

**目标**:定义对话的"原子",以及一个**不需要 API key 就能跑**的假模型,后续所有迭代都靠它驱动。

**对标 Claude Code**:`types/message.ts`(消息类型)+ `query/deps.ts`(依赖注入,模型可替换)。

**要写**:
- `message.py`:`Message` 类型。至少区分 `role`(user/assistant/system)、`content`;助手消息能携带"工具调用"列表。
- `model.py`:
  - `Model` 抽象基类,一个方法 `async complete(messages) -> AssistantMessage`。
  - `MockModel`:接收一个"脚本"(预设的回应序列),按顺序返回。**它让你无需真实 API 就能驱动整个编排。**

**做完的标准**:
- 能在一个 `main.py` 里:构造几条 Message → 喂给 MockModel → 打印它返回的助手消息。
- `uv run` 能跑起来。

**学到什么**:为什么 Claude Code 把"调模型"做成可注入的依赖(`deps.ts`)——**可测试、可离线跑编排**。面试考点:依赖注入。

---

### ⬜ M2 · 工具系统(Agent 的"手")

**目标**:让 Agent 能"做事"。定义工具抽象 + 注册表 + 一次运行的上下文。

**对标 Claude Code**:`Tool.ts`(工具抽象、`isConcurrencySafe`)+ `ToolUseContext`。

**要写**:
- `tool.py`:
  - `Tool` 抽象基类:`name`、`description`、`async run(input, ctx) -> str`。
  - **关键字段 `is_concurrency_safe`**:这个工具能不能和别的并发跑(只读=True,写=False)。这是 M3 的伏笔。
  - 一个简单注册表 `ToolRegistry`(按名字找工具)。
  - 两个示例工具:`ReadFileTool`(只读,safe=True)、`WriteFileTool`(写,safe=False)。
- `context.py`:`RunContext`——贯穿一次运行,至少含:`abort`(取消信号,用 `asyncio.Event` 或 flag)、`depth`(递归深度,M5 要用)。

**做完的标准**:
- 能手动构造一个工具调用 → `tool.run(...)` → 拿到结果字符串。
- 注册表能按名字取到工具。

**学到什么**:工具为什么要**自己声明**能不能并发(而不是调度器猜)——这是 Claude Code "声明式并发"的根。面试考点:function calling、关注点分离。

---

### ⬜ M3 · 工具编排:读并发 / 写独占 + 流式(并行的核心)

**目标**:模型一轮可能要调用多个工具——把它们**正确地并行/串行**地跑掉。这是整个项目**最该吃透**的一节。

**对标 Claude Code**:`services/tools/toolOrchestration.ts`(`partitionToolCalls` + 读并发 / 写独占 + AsyncGenerator)。**精确到行的讲解在 `docs/agent-orchestration.md` §2。**

**要写**:`orchestration.py`
- `partition_tool_calls(calls)`:按 `is_concurrency_safe` 把工具调用切成**连续批次**——连续的只读合成一批,任何写工具自己独占一批。
- `run_tools(calls, ctx)`:**异步生成器**。
  - 只读批 → `asyncio.gather` 并发跑(用 `asyncio.Semaphore(10)` 限并发,对标默认上限 10)。
  - 写批 → 一个一个 `await`(串行)。
  - **边完成边 `yield` 结果**(不是全跑完再返回)。

**做完的标准**:
- 喂一组工具调用 `[Read, Read, Write, Read]`,能打印出分批结果:`[Read,Read] [Write] [Read]`。
- 能观察到:两个 Read **同时**开始(加个 `asyncio.sleep` + 打时间戳验证),Write 单独排队。

**⚠️ 必踩的坑(故意让你踩)**:第一次你大概率会写出"明明并发却一个个跑"。原因通常是某处用了**阻塞调用**或忘了 `gather`。踩明白这个,你才真懂 async 并发。面试考点:asyncio 并发、并发安全。

**学到什么**:Claude Code"多 Agent 能并行"不是专门写的——是"Agent 工具被标记为 safe,落入只读批"的**副产品**(§2.2)。

---

### ⬜ M4 · 主循环(Agent 的"心脏":想→做→看)

**目标**:把 M1–M3 串成一个会自己转的 Agent。这是 Claude Code 的 `queryLoop`。

**对标 Claude Code**:`query.ts` 的 `queryLoop`(`while(true)`,§1)+ `maxTurns` 熔断(§2.6)。

**要写**:`loop.py`
- `async def run_loop(messages, model, tools, ctx, max_turns)` —— 异步生成器:
  ```
  while True:
      回应 = await model.complete(messages)     # 想
      yield 回应
      if 回应里没有工具调用: return            # 看:没活了,结束
      async for 结果 in run_tools(回应的工具调用, ctx):  # 做(用 M3)
          yield 结果
          把结果回灌进 messages                 # 结果塞回上下文 → 下一轮
      if 轮数 > max_turns: 注入"超限"消息; return  # 熔断
  ```

**做完的标准**:
- 用 MockModel 脚本化一段:"先调 ReadFile → 看到结果 → 再回答" → `run_loop` 能完整跑完,打印每一步。
- 故意让 MockModel 永远调工具,验证 `max_turns` 能刹住。

**学到什么**:这就是 **Prompt Chaining 的本质**——结果回灌成下一轮输入,链是循环的副产品。还有:为什么"自主"必须配 `max_turns`(防死循环烧钱)。面试考点:Agent loop / ReAct、安全闸。

> 🎉 **里程碑**:M4 做完,你有了一个**完整的单 Agent**,能想→做→看,能并行工具,能熔断。后面全是"让它生出更多自己"。

## 阶段二 · 多 Agent 协作

### ⬜ M5 · 子 Agent(递归 + 上下文隔离)

**目标**:让 Agent 能"派一个分身去办子任务"。**核心顿悟:子 Agent = 带隔离上下文,再调一次 `run_loop`。**

**对标 Claude Code**:`tools/AgentTool/runAgent.ts`(§3)。它结尾就是再调 `query()`。

**要写**:`subagent.py`
- 一个 `AgentTool`(它本身是个工具,`is_concurrency_safe=True` → 所以多个子 Agent 能并行!)。
- 它的 `run` 里:
  - **造一份隔离的上下文**(干净的 messages,只带子任务 prompt;`depth + 1`)。
  - **调 `run_loop`**(就是 M4 那个,递归!)。
  - 只把子 Agent 的**最终结论**返回给父级(不是全过程)。
- 加一个**递归守卫**:`depth` 超过上限就拒绝再派(防无限套娃)。

**做完的标准**:
- 主 Agent 调用 AgentTool → 子 Agent 独立跑完一个 mini 循环 → 把结论交回 → 主 Agent 继续。
- 因为 AgentTool 是 safe 的,派两个子 Agent 能在 M3 的并发批里**同时跑**。

**学到什么**:整个多 Agent 不需要新框架——**就是主循环的递归复用**(§0 核心心智模型)。上下文隔离=保护父级"脑容量"。这里也涌现了 **Routing**(选派哪个子 Agent)。面试考点:递归编排、上下文隔离/预算。

---

### ⬜ M6 · 协调器(Orchestrator-Workers)

**目标**:一个 leader 只负责"派活 + 综合",worker 干活,worker 工具被收窄。

**对标 Claude Code**:`coordinator/coordinatorMode.ts`(§4.5-A)。

**要写**:`coordinator.py`
- 一个"coordinator 模式"开关。开启后:
  - **leader 的 system prompt** 改成"你只派活和综合,不自己下场"。
  - **worker 的工具被收窄**(白名单):比如不给 worker 再派子 Agent 的权力(对标 `INTERNAL_WORKER_TOOLS` 被移除)。
  - worker 的结果**包装成"伪用户消息"**回灌给 leader(对标 `<task-notification>`,§4.5-B)——leader 当成"又收到一条用户消息"来处理。
- 用 MockModel 脚本化一个场景:leader 派 2 个 worker 并行研究 → 收到两份结论 → 综合成一个回答。

**做完的标准**:
- 能看到完整链路:leader 派活 → 2 worker 并行 → 结论回灌 → leader 综合。
- worker 试图调"派子 Agent"工具时被拒(白名单生效)。

**学到什么**:这就是教科书的 **Orchestrator-Workers**。关键设计:**编排权集中在 leader,worker 只干活**;异步结果伪装成用户消息→无缝融进主循环。面试考点:编排器模式、任务分解。

---

### ⬜ M7 · Agent 间通信(mailbox + SendMessage)

**目标**:让 Agent 之间能**直接传话**(不只是 leader↔worker 的单向回灌)。

**对标 Claude Code**:`utils/mailbox.ts` + `tools/SendMessageTool/`(§4.5-C)。

**要写**:
- `mailbox.py`:一个极简异步队列(对标 73 行的 `mailbox.ts`):
  - `send(msg)`:有人在等就直接唤醒,否则入队。
  - `receive(filter)`:取一条匹配的;没有就**挂起等待**(返回一个 await 的东西)。
  - `poll(filter)`:非阻塞瞅一眼。
- `send_message` 工具:Agent 调它给另一个 Agent 发消息。**强约束:Agent 的普通输出别人看不见,要沟通必须调这个工具。**

**做完的标准**:
- 两个 Agent:A `send` 一条 → B 正在 `receive` 挂起 → 被唤醒收到。
- 跑通一个"A 让 B 帮查个东西,B 查完回传给 A"的最小协作。

**学到什么**:通信机制本身极简(就一个队列),复杂度在"谁能给谁发"的上层。对比 M6:**Coordinator 是星型(单向回灌),Team 是网状(双向通信)** —— 两种协作拓扑。面试考点:异步消息队列、生产者-消费者。

> 🎉 **里程碑**:M7 做完,你已完整复刻了 Claude Code 的**两套多 Agent 协作模型**。编排内核完成。

---

### ⬜ M8 · 评估-优化循环(Evaluator-Optimizer)

**目标**:补齐五大模式最后一个。**不是新机制,是用现有零件组合。**

**要写**:一个 `examples/evaluator_optimizer.py`
- 用现有零件拼:worker_A 产出草稿 → worker_B 批判打分 → 不达标就把批判回灌给 worker_A 重做 → 循环直到达标或到上限。

**做完的标准**:能看到"产出→批判→改进→再批判"收敛的过程。

**学到什么**:复杂模式都是**基础零件的组合**,不需要新框架。面试考点:自我批判、迭代收敛、防止无限循环。

## 阶段三 · 接真实模型 + 落地业务(老外挖中国供应商)

### ⬜ M9 · 接入真实 LLM(多 provider,可灵活切换)

**目标**:把 MockModel 换成真实模型,且**从一开始就支持多家**:Claude / GPT / 国内模型,用户可随时切换。开发阶段先只接你本地的 **Claude via AWS Bedrock**。

**为什么现在就要管多 provider**:这是**架构决策,不能拖到做产品再补**。但代价几乎为零——因为 M1 的 `Model` 抽象已经把"调模型"隔离了,加一个 provider = 加一个类。这正是对标 Claude Code `deps.ts` 依赖注入的回报。

**要写**:`providers/` 子包,每个 provider 一个类,**都实现 M1 的同一个 `Model` 抽象**:
- `providers/bedrock.py` → `BedrockClaudeModel`(**开发阶段先做这个**,走 AWS Bedrock 接 Claude)
- `providers/anthropic.py` → `AnthropicModel`(Anthropic 官方 API,后续)
- `providers/openai.py` → `OpenAIModel`(GPT,后续)
- `providers/qwen.py` 等 → 国内模型(通义/DeepSeek/智谱…,多数兼容 OpenAI 协议,可复用 OpenAIModel 改 base_url)
- `providers/__init__.py` → 一个 `make_model(provider, model_name, **opts)` 工厂 + 配置读取(从 env / 配置文件选 provider)

**关键设计原则**(让切换真的"灵活"):
- **统一接口**:无论哪家,编排内核只认 M1 的 `Model.complete(messages) -> AssistantMessage`。provider 内部负责把我们的 Message ↔ 各家 API 格式互转。
- **工具调用归一化**:各家的 function/tool calling 格式不同(Anthropic 的 `tool_use` / OpenAI 的 `tool_calls`),provider 层要**翻译成我们 M1 统一的 ToolCall 结构**。这是多 provider 最容易踩的地方,集中在 provider 层处理,内核不感知。
- **能力差异兜底**:有的国内模型不支持原生 tool calling → provider 层可降级(如用 prompt 模拟),或在 `make_model` 时标注能力,让上层选择。

**做完的标准**:
- 同一段编排代码,改一个配置项(`provider=bedrock` → `provider=openai`),不改任何编排逻辑,就能换模型跑通。
- 开发阶段:`BedrockClaudeModel` 能用你的 AWS 凭证真实调通 Claude。

**学到什么**:为什么"调模型"必须是可替换依赖。多 provider 适配的真正难点不是"调通 API",而是**把各家不同的消息/工具格式归一化到一个内部表示**——这是做 AI 产品的核心工程能力,也是高频面试点。

> 依赖:AWS Bedrock 用 `boto3` 或 `anthropic[bedrock]` SDK;OpenAI 兼容的用 `openai` SDK。到这步再 `uv add`,别提前装。

### ⬜ M10 · 调研工具 + 溯源/防幻觉
**目标**:给 Agent 真实的"手":搜索、抓网页。**每条结论必须带来源;查不到就说查不到。**
**要写**:`SearchTool`、`FetchTool`;结果结构里强制带 `source` 字段。
> 这是你产品差异化的核心(中外交叉空白),也是面试最值钱的 RAG/防幻觉考点。

### ⬜ M11 · 业务编排:供应商挖掘 + 画像
**目标**:用 M6 的协调器 + M5 的子 Agent,搭你的真实场景。
**要写**:Coordinator 派:挖掘 worker(找供应商)→ 背景 worker(扒资料)→ 翻译解读 worker(中文→英文+解读)→ 匹配 worker(打分)→ 综合成一张带来源的供应商画像表。
> 这就是"皮肤"。内核(M1–M8)通用,以后换"竞品分析""市场调研"都只换这一层。
