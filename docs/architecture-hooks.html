Hooks · How Claude Code Hooks Collaborate

Hook 协作专题 · Claude Code 的钩子如何参与 Agent 运行

Hook 是什么:一个 hook = 一段你登记在 ~/.claude/settings.json 里的外部命令/规则,绑定到 Agent 运行的某个事件(如 PreToolUse)。Harness 跑到那个节点时自动调用它,并按它的返回值决定放行 / deny 阻止 / 注入 additionalContext。

它解决了什么问题:没有 hook 时,想加"写文件前先审一道""改完代码自动 lint""危险命令拦截""会话开始注入项目规范"这类行为,只能改 Agent 内核代码——每个团队需求不同,内核会被改得千疮百孔。Hook 把这些可变策略从内核里剥出来,挪到外部配置:内核保持通用稳定,各团队/项目在配置里按需挂自己的规则。一句话——它是"不改源码就能定制 Agent 行为"的官方扩展点(等价于 Web 框架的 middleware、Git 的 pre-commit hook)。

下图分两块:① 上半 — 最常用的工具/会话类 hook,沿一条 ReAct 时间线摆开(红框=在该步拦截);② 下半 — HOOK_EVENTS 的 全部 29 个事件按 6 类分组(coreTypes.ts),外加 4 种 hook 类型、matcher 过滤、返回决策这三组机制。事件名、字段名均来自真实源码。

一句话读懂:把 Agent 跑一轮想象成流水线(①→⑤),Hook 就是装在特定工位上的"质检/拦截开关"。最常用的是工具前后(PreToolUse 拦危险操作 / PostToolUse 自动格式化),其余覆盖输入、收尾、压缩、会话生命周期。

它怎么"协作":Harness 跑到某工位 → 触发对应事件 → 调用你登记的 hook → 看返回值:放行就继续、deny就拦下这一步、additionalContext就把信息塞进对话。整个过程不改主循环一行代码——hook 全在 ~/.claude/settings.json 外部配置里登记(机制实现见 utils/hooks.ts,5022 行)。

规模真相:Claude Code 的 HOOK_EVENTS 实际有 29 个(见上方图 ② 的 6 类分组),本图时间线只画了最常用的一组;hook 还支持 4 种执行类型(command / prompt / http / agent)、matcher 过滤、多种返回决策。这也说明:核心机制"在节点上回调外部 hook、按返回值决定流程"很简单,大量代码是覆盖各种事件与边界的生产外壳(utils/hooks.ts 5022 行)。
