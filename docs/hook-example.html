Hook · Worked Example

Hook 到底怎么和 Agent 配合 · 一个场景,从头追到尾

光看"29 个事件"记不住。这里只用一个真实场景,把 hook 怎么"插一脚、改变 Agent 走向"完整演一遍。看完你就懂了:hook = Agent 干活到某一步时,先停下来问一句"我登记的脚本同不同意",再按回答决定下一步。

场景
你对 Claude 说:"帮我把数据库密码写进 config.py"

你提前在 ~/.claude/settings.json 里登记了两个 hook(下面这段)。注意:你只是写了配置,没碰任何 Agent 代码。

~/.claude/settings.json（你登记的两个 hook）
"hooks": {
  // ① 写文件【前】:跑 guard-secrets.sh 审一道
  "PreToolUse": [{
    "matcher": "Write",            // 只在调用 Write 工具时触发
    "hooks": [{ "type":"command", "command":"./guard-secrets.sh" }]
  }],
  // ② 写文件【后】:自动跑格式化
  "PostToolUse": [{
    "matcher": "Write",
    "hooks": [{ "type":"command", "command":"black ." }]
  }]
}
1

MODEL 模型决定:调 Write 工具写 config.py

Claude 读懂你的话,产出一个工具调用:Write(file_path="config.py", content="DB_PASSWORD=hunter2")。这时工具还没真正执行。

2

HARNESS 到了 PreToolUse 节点 → 触发 hook

Harness 在真正写文件之前停一下,发现你登记了 PreToolUse + matcher:"Write",匹配!于是把当前信息打包成 JSON,通过标准输入喂给 guard-secrets.sh:

Harness → hook 脚本（stdin 收到的输入）
{
  "hook_event_name": "PreToolUse",
  "tool_name": "Write",
  "tool_input": {
    "file_path": "config.py",
    "content": "DB_PASSWORD=hunter2"
  }
}
3

HOOK 脚本判断:明文密码写进 .py?不行!

guard-secrets.sh 检查到 content 里有 DB_PASSWORD= 且目标是 .py 文件,打印一段 JSON 到标准输出,裁决 deny(拒绝):

hook 脚本 → Harness（stdout 返回的决策）
{
  "hookSpecificOutput": {
    "permissionDecision": "deny",
    "permissionDecisionReason":
      "禁止把明文密码写进 .py,请改用环境变量"
  }
}
⛔ 裁决 = deny → 这次 Write 被拦下,不执行
4

MODEL 模型收到"被拒 + 原因",换个方案

关键就在这:被拦下的原因会作为观察结果回灌给模型。模型"看到"了 "禁止明文密码,请改用环境变量",于是自己改主意——不写 config.py 了,改成:① 把密码写进 .env;② 让 config.py 从环境变量读。

↑ 这就是 hook 最妙的地方:它没有"报错崩溃",而是把约束变成反馈,让 Agent 自己绕道。

5

HARNESS 新的 Write(.env) 又触发 PreToolUse

模型这次调 Write(".env", "DB_PASSWORD=hunter2")。Harness 又把它喂给 guard-secrets.sh。脚本规则是".env 文件允许放密码",所以这次放行:

hook 脚本 → Harness
{ "hookSpecificOutput": { "permissionDecision": "allow" } }
✅ 裁决 = allow → Write 正常执行,.env 写好了
6

HOOK 写完后,PostToolUse 自动跑格式化

文件写完,Harness 到 PostToolUse 节点,触发第二个 hook → 跑 black . 把代码格式化。它不改流程,只做"善后",跑完就继续。

7

MODEL 任务完成,回答你

Claude 告诉你:"已把密码放进 .env,并让 config.py 从环境变量读取(没有把明文写进代码)。" —— 你想要的安全约束,全程没改一行 Agent 代码,只靠两个 hook 实现了。

读完这一个例子,"怎么配合"就清楚了:

① 你登记规则(settings.json),不碰 Agent 代码;
② Agent 干活跑到某个节点(如"工具执行前"),Harness 自动停一下,把现场打包成 JSON 发给你的 hook;
③ hook 返回 JSON 裁决:deny 拦下 / allow 放行 / additionalContext 塞信息;
④ 被拦下时,原因会回灌给模型,模型自己绕道重试 —— 不是崩溃,是"协作"。

一句话:hook 就是你给 Agent 装的"红绿灯 + 摄像头"——它跑到路口先看灯(你的脚本),灯说停就停、说走就走,而且为什么停会告诉司机(模型),让它换条路。29 个事件 = 29 个可以装红绿灯的路口,但机制全是这一套。
