# Hook 是什么

## 一个你大概率遇到过的小烦

你在用 AI Agent 写代码，顺手在 CLAUDE.md 里立了几条规矩：改完代码记得跑 lint、删东西前先问我一声、commit 信息写中文。

头几次它都照办，你还挺欣慰。可干着干着——对话越堆越长，它的注意力被一堆别的事占着——某一次就漏了：该跑的 lint 没跑，该问的没问，commit 又蹦出一句英文。你不是没写清楚，它也不是故意的，就是……忘了。

这事说大不大，烦在它**不稳定**。你真正想要的，从来不是"它大概率会做"，而是"它一定会做"。而这种需求其实到处都是——

- 改完代码，自动跑一遍格式化和 lint，别等 CI 才飘红；
- 危险命令（`rm -rf`、`git push --force`）真要执行前，先拦一道；
- 每次开工，先把团队规范、手头在做的事喂给它；
- 活没干完（测试还没过）就不许它说"我做完了"。

它们长得各不一样，可要的是同一件事：**在某个确切的时刻，自动地、必然地，发生一件事。**

## 那……写进 CLAUDE.md 不就行了？

这是几乎所有人的第一反应，也是最该先讲清的一层。

而你刚才已经试过了——把规矩写进 CLAUDE.md，结果"有时听、有时忘"。问题不在你写得不够清楚，而在 CLAUDE.md（还有提示词、记忆）说到底是**写给模型看的建议**：它读到了，然后"自己决定"要不要照做。是建议，就带概率——这次照做，下次就可能漏。

所以规范、偏好、背景这类**软的、劝导性的**东西，交给 CLAUDE.md 正合适。可一旦你要的是**保证**——这条命令必须被拦下、这一步必须被执行——光靠模型自觉就不够了。

说白了，它俩不是一个东西，也不是谁替代谁：

| | 谁来执行 | 性质 | 管什么 |
| :-- | :-- | :-- | :-- |
| CLAUDE.md / 提示词 / 记忆 | 模型（读完自己决定做不做） | 带概率，是劝 | 规范、偏好、背景 |
| Hook | 引擎（到点照办，不商量） | 确定，是强制 | 必须发生 / 必须拦下的动作 |

差别全在第二列：Hook 是**跑 Agent 的那个引擎**去执行的，根本不过模型那道"愿不愿意"。这才是它能给你"保证"的原因。

## 那直接改源码、加几个 if 呢？

既然得让引擎来管，最直接的念头就是：改 Agent 源码，在该拦的地方塞几个 `if`。

但这条路很快就堵死。你要拦的和我要拦的根本不一样，这些 `if` 全堆进内核，没几天就没人敢动了；而且你改过的内核，没法分发给别人，也没法跟着官方一起升级。软件工程里的老教训：通用的核心要稳，会变的策略别往里塞，不然核心迟早被拖烂。

反过来才对——核心保持干净，把会变的那部分请出去，放到一个"它能回头来问、但自己不掺和"的地方。

这地方，就是 Hook。

## 它是什么

把 Agent 想成一个能干但你管不住嘴的实习生。你在他必经的路口架关卡、贴纸条：要执行命令时"先看是不是 rm -rf，是就拦下"。他每次走到路口都得停下，让关卡按纸条办事，再决定放不放过。

说白了：**一个 Hook 就是你登记的一条规则，绑在 Agent 运行的某个事件上，引擎跑到就自动停下执行它，按答复决定怎么走。** 你不碰内核，只在路口贴纸条（写配置）。那张"纸条"长这样：

```jsonc
{
  "hooks": {
    "PreToolUse": [                    // 路口：工具执行前
      { "matcher": "Bash",             // 只在用 Bash 时拦
        "hooks": [
          { "type": "command",         // 关卡干啥：跑个脚本
            "command": "./check-danger.sh" }
        ]
      }
    ]
  }
}
```

读出来就是：每次 Agent 想用 Bash 之前，先跑一下 `check-danger.sh`。这跟 git 的 pre-commit、Web 的中间件是一回事：不改核心，在关键节点插一手。

这里 `type` 不止"跑脚本"一种，共五种：`command`（跑 shell）、`http`（把事件 POST 给你的服务）、`mcp_tool`（调一个已连接的 MCP 工具）、`prompt`（让小模型做个是非判断）、`agent`（派个能读文件的子 Agent 去核实再决定）。

`matcher` 是粗筛：工具类事件按工具名匹配（`Bash`、`Edit|Write`、`mcp__.*` 匹配某 MCP 服务器的全部工具），会话类按"怎么开始的"（`startup`/`resume`/`compact`），等等；不写就每次都触发。想更细就再加 `if`，按工具名加参数一起匹配，比如 `Bash(git *)` 只在跑 git 时触发、`Edit(*.ts)` 只对 TS 文件触发。

## 它怎么工作

四步。一、跑到路口停下，把情况打个包给规则：什么事件、哪个工具、什么参数。二、筛出该跑的规则并行跑，各有超时，一条卡住不拖累别的。三、读答复：退出码 0 放行、2 拦下，想精细控制就吐一段 JSON。四、按判决走，四种——拦下、放行（也能顺手改参数）、注入（往对话里塞段话给它看）、喊停。

单说一点：Hook 拦下时不是抛错崩溃，而是把"我拦你了，因为……"喂回模型，模型读到就自己改——不让删那就移到回收站。拦下不是终点，是一次返工。

## 能架在哪 · 多 Agent 里的位置

顺着一次会话走：开始、你输入、工具前、工具后、收尾、结束，二十多个时刻都能架，最常用是工具前后。正因为每层都能架，Hook 不属于某个模块，而是像一根线穿过整个系统——这叫横切机制，跟日志、权限一类。

多 Agent 里它还多一层：主 Agent 每派生一个子 Agent，先过"启动"关卡（可叮嘱它"只看安全问题"），子 Agent 内部调工具照样过关卡，而且关卡分得清现在是谁在干活，作用域随上下文隔离。

## 凭什么敢用

不信任的项目一律不跑（防被陌生仓库里的恶意 Hook 黑）；没挂就零开销；拦截是软拦——变成反馈而不是崩溃。
